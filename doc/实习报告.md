# 基于分布式流处理平台的宝可梦对战数据分析系统

## 1. 实习目的与意义

本次生产实习旨在深入理解和实践大数据分布式处理技术，通过构建"宝可梦对战数据分析系统"这一完整工程，掌握以下核心技术栈：

1. 分布式消息队列 Kafka 的部署、配置与生产消费模式
2. Apache Spark Structured Streaming 实时流处理框架的应用
3. Vue 3 前端框架与 Python FastAPI 后端的数据可视化实现
4. SQLite 数据库的实时查询优化与缓存策略
5. 多节点 Linux 集群（ZooKeeper + Kafka + Spark）的运维管理

通过该项目的完整开发周期，将课堂所学的分布式计算、数据库原理、前后端开发等理论知识转化为实际工程能力，为从事大数据分析岗位奠定基础。

## 2. 实习任务

1. **数据分析引擎搭建**：设计并实现 SQLite 持久化连接 + TTL 缓存的分析查询引擎，支持精灵使用率、招式排行、道具排行、特性排行、属性分布、同队搭配、对位胜率等多维度统计分析
2. **实时数据管道构建**：搭建 Kafka → Spark Structured Streaming → SQLite 的端到端实时数据处理管道，实现数据从产生到展示的全链路自动化
3. **前端可视化开发**：基于 Vue 3 实现分析仪表盘，包含排行榜 FLIP 动画、精灵详情钻取、对位胜率 PK 界面、同队搭配图表等交互功能
4. **集群环境部署**：在三节点 Ubuntu 虚拟机组上部署 ZooKeeper（3节点 Quorum）+ Kafka + Spark 集群，实现分布式数据处理
5. **前后端联调**：完成 Windows 物理机前端与 VM 集群后端 API 的跨网络通信，确保数据流畅展示

## 3. 实习内容

### 3.1 需求分析

- 宝可梦Gen9对战数据的实时采集与处理
- 精灵使用率、招式、道具、特性等多维度排行
- 实时数据推送与前端平滑动画
- 同队搭配分析、对位胜率查询
- 支持本地单机（SQLite）与集群（Kafka + Spark）双模式运行

### 3.2 概要设计

系统架构分为三层：

```
┌─ 展示层 (Windows) ──────── Vue 3 前端 ────────────┐
│  DataTable / HeadToHeadChart / TeamSynergy        │
├─ 服务层 (VM: cyc) ──────── FastAPI :8000 ─────────┤
│  sql_analytics.py / battle_importer               │
├─ 数据层 (VM: myz/cyc/lzx) ─ Kafka + Spark ────────┤
│  gen_battle_stream → Kafka → Spark → SQLite       │
└───────────────────────────────────────────────────┘
```

### 3.3 详细设计

**数据模型**：battle_pokemon_states 表，包含 battle_id、species_id、hp_pct、fainted、ability_id、item_id、move_ids、side_index、turn 等字段。

**API 路由**：`/stats/deep/summary`、`/stats/deep/meta`、`/stats/deep/head-to-head`、`/stats/deep/team-synergy`、`/cluster/gen/start` 等 15+ 端点。

**前端组件**：DataTable（排行榜，支持 FLIP 动画）、HeadToHeadChart（对位胜率）、TeamSynergyChart（同队搭配）、TypePieChart（属性分布）。

## 5. 项目开发过程

### 5.1 第一阶段 —— 本地单机版本

- SQLite 三层数据库分离（pokemon.db 参考库 + output.db 历史库 + output_recent.db 实时窗口）
- 文件监听器（watcher）实时扫描 battle_logs 目录，自动导入新对战
- 精灵详情钻取（点击精灵 → 查看其招式/道具/特性）


### 5.2 第二阶段 —— 前端可视化与动画

- Vue 3 TransitionGroup FLIP 动画实现排行榜平滑滑动
- 双表架构（表A：filtered 数据正确 → 表B：_display 动画缓冲）
- 幽灵占位（ghost slots）保证固定表格高度
- 状态机动画控制（播放 0.5s → 暂停 0.5s → 下一行）
- WebSocket 实时推送 + 10秒定时轮询
- 对位胜率 PK 界面、同队搭配图表

### 5.3 第三阶段 —— 分布式集群部署

- 三节点 ZooKeeper Quorum 集群搭建（myz + cyc + lzx）
- Kafka 集群部署于 myz 节点，Topic 3分区
- Spark Master（myz）+ Worker（cyc, lzx）分布式计算
- 数据管道：gen_battle_stream → Kafka → Spark Streaming → SQLite → API

### 5.4 第四阶段 —— 双模式兼容与系统联调

- 环境变量 POKEMON_MODE=local/cluster 实现双模式切换
- Windows 前端通过 Vite 代理跨网络连接 VM API（192.168.209.137:8000）
- 前端远程控制产数（API `/cluster/gen/start|stop`）
- WebSocket cluster broadcast 实现集群模式实时推送
- Kafka 数据积压排查与 topic 清理

### 6.2 遇到的问题与解决方案

1. **Kafka 连接超时**：advertised.listeners 配置错误导致客户端重定向到不可达IP → 配置为正确的 Tailscale 虚拟网络 IP
2. **ZooKeeper 单点故障**：单节点无法形成 quorum，Kafka 启动失败 → 确保至少2/3节点在线
3. **动画数据错误**：静态计算 newIdx 导致中间态排序错误 → 改为 FLIP 一步到位
4. **Kafka 数据积压**：重复消费导致精灵出场数暴涨 → 删除日志文件重建 topic
5. **集群模式下前端无数据**：WebSocket 连接超时 → 修复 Vite 代理配置，VITE_API_HOST 默认值
